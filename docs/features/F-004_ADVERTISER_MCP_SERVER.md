# F-004: Advertiser MCP Server

**Status:** Open
**Impact:** Expose the Advertiser model via MCP (Model Context Protocol) so agents can navigate and modify advertiser data. Mount at `/mcp`. No authentication — public access.

## Problem

Agents (MCP clients) need to interact with advertiser data — list, retrieve, create, update, delete — through the Model Context Protocol. The agent must understand the Advertiser model structure and how to use the available tools.

## Solution

### Overview

- Use **django-mcp-server** to expose Advertiser as MCP tools.
- Mount the MCP endpoint at **`/mcp`** (base URL).
- Provide **context/instructions** so the agent understands the Advertiser model and API usage.
- No security: endpoint is public; anyone can use it.

### MCP Endpoint

- **URL:** `{host}/mcp` (e.g. `http://localhost:8000/mcp` — no trailing slash for streamable HTTP)
- **Transport:** Streamable HTTP (MCP spec)
- **Authentication:** None

### Advertiser Model Context (for agent instructions)

The agent must understand:

| Field | Type | Description |
|-------|------|-------------|
| `id` | int | Primary key |
| `name` | str | Advertiser name |
| `discount` | Decimal | Discount rate (0–1, e.g. 0.05 = 5%) |
| `default_conversion_rate` | Decimal | Default conversion rate (e.g. 0.01) |
| `monetization_type` | enum | `CPC` or `CPA` |
| `billing_source` | enum | `INTERNAL` or `PARTNER` |
| `billing_timezone` | enum | `UTC`, `EST`, `CST`, `PST` |
| `created_at` | datetime | Creation timestamp |
| `updated_at` | datetime | Last update timestamp |

**Operations:** List advertisers (with optional filters), retrieve by id, create, update, delete.

### Implementation Approach

1. **ModelQueryToolset** for Advertiser — list and query (read). Inherits filters from F-003 if present on `AdvertiserViewSet`.
2. **DRF publish annotations** — expose `AdvertiserViewSet` create/update/delete as MCP tools (`drf_publish_create_mcp_tool`, `drf_publish_update_mcp_tool`, `drf_publish_delete_mcp_tool`, `drf_publish_list_mcp_tool`).
3. **DJANGO_MCP_GLOBAL_SERVER_CONFIG** — set `instructions` with context about the Advertiser model and how to use the tools.
4. **Tool names** — generated by django-mcp-server from model/view. Observed names: `AdvertiserViewSet_ListTool`, `AdvertiserViewSet_CreateTool`, `AdvertiserViewSet_UpdateTool`, `AdvertiserViewSet_DeleteTool`, `query_data_collections`, `get_server_instructions`. Verify via `python manage.py mcp_inspect`.

### Dependencies

- **F-001:** Advertiser API must exist.
- **django-mcp-server:** Already in `pyproject.toml`. See `pyproject.toml` for version.

### Known Issues

| Issue | Symptom | Workaround / Action |
|-------|---------|---------------------|
| **DRF patch** | `AttributeError: 'WSGIRequest' object has no attribute 'original_request'` in `mcp_server/djangomcp.py` when non-MCP requests hit DRF views. Unit tests and REST API (Httpie) return 500. | Check `hasattr(request, 'original_request')` before use; apply patch only for MCP requests; or report to django-mcp-server. |
| **MCP CreateTool args** | CreateTool returns `{'name': [ErrorDetail(string='This field is required.', code='required')]}` even when `body: {name: '...'}` is passed. | Verify argument schema: django-mcp-server may expect a different structure (e.g. flat args vs nested `body`). Inspect tool schema via `mcp_inspect` and adjust call format. |

## Agent Context (Instructions Template)

Use in `DJANGO_MCP_GLOBAL_SERVER_CONFIG["instructions"]` and tool docstrings:

```
Advertiser model: Represents an advertiser (job poster). Fields: id, name, discount (0–1), default_conversion_rate, monetization_type (CPC|CPA), billing_source (INTERNAL|PARTNER), billing_timezone (UTC|EST|CST|PST), created_at, updated_at.

Use the list tool to list advertisers (supports filters). Use create to add a new advertiser. Use retrieve to get one by id. Use update to modify an existing advertiser. Use delete to remove one.
```

## Httpie Examples (Manual / E2E)

MCP uses a different protocol (streamable HTTP) than REST. Use **MCP Inspector** or **Python MCP SDK** instead of Httpie for MCP e2e. For REST API verification (same underlying data), use:

```bash
# List advertisers — 200, list (verify REST API still works)
http GET ":8000/api/v1/advertisers/"

# Create — 201 (verify REST API still works)
http POST ":8000/api/v1/advertisers/" name="MCP Test"

# MCP endpoint: connect with MCP Inspector to http://localhost:8000/mcp
# Call tools: AdvertiserViewSet_ListTool, AdvertiserViewSet_CreateTool, etc. (see Implementation Approach)
```

**Expected REST response (200 OK, list):**

```json
[
  {
    "id": 1,
    "name": "MCP Test",
    "discount": "0.0000",
    "default_conversion_rate": "0.0100",
    "monetization_type": "CPC",
    "billing_source": "INTERNAL",
    "billing_timezone": "UTC",
    "created_at": "2025-03-05T12:00:00Z",
    "updated_at": "2025-03-05T12:00:00Z"
  }
]
```

**MCP SDK Python example (E2E):**

```python
from mcp.client.streamable_http import streamablehttp_client
from mcp import ClientSession
import asyncio

async def main():
    async with streamablehttp_client("http://localhost:8000/mcp") as (read_stream, write_stream, _):
        async with ClientSession(read_stream, write_stream) as session:
            await session.initialize()
            # List (works)
            result = await session.call_tool("AdvertiserViewSet_ListTool", {})
            print(result)
            # Create: check schema via mcp_inspect — args may need flat format, not nested body
            # result = await session.call_tool("AdvertiserViewSet_CreateTool", {"name": "Test"})
            # If "name required" error, inspect tool input schema and adjust.

if __name__ == "__main__":
    asyncio.run(main())
```
